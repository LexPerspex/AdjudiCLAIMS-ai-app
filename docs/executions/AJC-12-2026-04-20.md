## Baseline Scenarios Implemented

### Scenario 1: Chat Endpoint Latency

| Percentile | Threshold | Rationale |
|------------|-----------|-----------|
| p50 | 2,000 ms | Median must feel fast for routine queries |
| p95 | 5,000 ms | Acceptable for Claude API + RAG round-trip |
| p99 | 10,000 ms | Cold starts + large document context edge cases |

### Scenario 2: Document Processing Throughput

| Percentile | Threshold | Rationale |
|------------|-----------|-----------|
| p50 | 15,000 ms | Typical single-page medical report via Document AI |
| p95 | 30,000 ms | Multi-page reports up to 20 pages |
| p99 | 60,000 ms | Large documents (50+ pages), async processing tail |

### Scenario 3: Concurrent Users (10 users / 30s)

| Metric | Threshold | Rationale |
|--------|-----------|-----------|
| Concurrency | 10 users | Per ticket requirement |
| Duration | 30 seconds | Per ticket requirement |

## Design Decisions

1. **No new dependencies** — Used native `fetch` + `tsx` (already in devDependencies). Autocannon/k6 would add complexity without proportional value for this project's scale.

2. **Baselines as assertions** — Thresholds in `baselines.json` are enforced at the Vitest layer (always runs on CI) and at the smoke test layer (post-deployment). Changing a threshold requires a PR, creating an audit trail.

3. **Live-server caveat documented** — Chat and document latency tests measure the auth-gate fast path without a running server. Full AI latency measurement requires `DEPLOYMENT_URL` + a valid session token. This is documented in `tests/performance/README.md` and in the test file itself.

4. **Percentile math unit-tested** — The `percentile()` helper is independently tested to prevent regressions in the statistical calculations.
