### Why we did not re-implement

1. **Avoid regression.** The 90af9d5 versions are substantive: each step
   carries a real description, a CA statutory authority (LC / 8 CCR / 10 CCR
   / Ins. Code), and a compliance rationale. Re-authoring would risk losing
   field-tested content for no benefit.
2. **No spec gap identified.** Spot-check of Workflow 12 (DOI Audit Response)
   against the spec doc shows the implementation captures the spec's intent:
   six steps covering scope identification, file assembly, self-check,
   production, response to findings, and appeal.
3. **Tests already cover structure.** `tests/unit/workflows-phase10.test.ts`
   already validates the 20-workflow set for required fields, unique IDs,
   ≥4 steps per workflow, valid `uplZone`, and skip-reason consistency
   (204 tests).

## Delta on this branch

### `tests/unit/workflows-citation-integrity.test.ts` (new — 103 tests)

Adds a tighter regression net on top of `workflows-phase10.test.ts`:

- **Spec-ID alignment** — asserts the exported workflow IDs exactly match the
  20 IDs defined in the spec doc (catches accidental rename/drop).
- **Citation rigor** — asserts every workflow's top-level `authority` and
  every step's `authority` cite a recognised CA statutory source
  (`LC \d`, `8 CCR \d`, `10 CCR \d`, `Ins. Code \d`, `MTUS`, `OMFS`) OR a
  bounded operational allowlist (`Carrier ...`, `Internal claim handling
  protocol`, `No specific statutory deadline`).
- **Substance check** — every step description ≥60 chars, every compliance
  note ≥40 chars, no `TODO`/`TBD`/`FIXME` strings (catches future drive-by
  stubbing of a workflow step).
- **UPL invariant** — no workflow may be RED-zoned. Workflows are GREEN or
  YELLOW only; RED is reserved for chat queries that get blocked, not for
  examiner-facing decision aids.
- **Skip consistency** — skippable steps must have a meaningful `skipReason`
  (≥20 chars); non-skippable steps must NOT have one (catches data drift in
  either direction).

## Decisions & deferrals

- **Did not extend frontend.** The generic workflow rendering engine already
  consumes `WORKFLOW_DEFINITIONS`; new workflows are picked up automatically.
  No UI changes needed.
- **Did not write a per-workflow happy-path execution test.** The spec
  intent is "validate factual / structural correctness", which the data-level
  tests cover. End-to-end traversal would require simulating a claim and a
  user — better captured in the existing `workflow-engine.test.ts` rather
  than re-built here.
- **No items deferred to ISSUES.md** — no follow-ups identified during the
  audit.
